# ADR-0002 — Code is the single source of truth; Figma is generated downstream

**Status:** Accepted  
**Date:** 2026-06-24

## Context

Design tokens can flow in two directions:

1. **Code → Figma** — tokens authored in code, generated *into* Figma Variables. Figma consumes them.
2. **Figma → code** — tokens (and even component code) authored in Figma, exported/generated *out* to code (Figma Dev Mode MCP, "make code").

Supporting both directions as authoritative creates two sources of truth that drift apart. A portfolio whose thesis is "I build maintainable systems" cannot itself have an ambiguous source of truth.

## Decision

**Code is the single source of truth.** Token files (DTCG format) are authored in the repo. Style Dictionary builds them into CSS custom properties for the site, and a one-directional sync (Tokens Studio / a Figma plugin or MCP) pushes the same tokens *into* Figma Variables.

Figma is an **output**, never a generator of site code. The "Figma → code" direction is explicitly not used.

## Consequences

- The token pipeline runs one way: `tokens/*.json (DTCG)` → Style Dictionary → CSS variables (site) **and** → Figma Variables (design artifact).
- Figma stays in sync by re-running the sync after token changes; designers see the canonical tokens but the repo owns them.
- Component code is authored by Leonid + AI (orchestrator workflow), not generated from Figma. This is deliberate: the Track C value proposition is that *he* architects the system.
- Manual CSS authoring is minimized — token CSS is generated by Style Dictionary, not hand-written — without surrendering authorship of the system itself.

## Alternatives considered

- **Figma → code as source of truth** — rejected: produces throwaway scaffolding, not a maintainable token architecture, and undercuts the "I build the system" story.
- **Bidirectional sync** — rejected: two sources of truth, guaranteed drift, contradicts the brief's own "design and implementation stay aligned" goal.
